# Best Practices ‚Äî Public Repositories

Reference document for git-sentinel. Defines what the tool enforces, recommends, and generates for public GitHub repositories.

---

## Branch Strategy

| Branch | Purpose | Created from | Merges into |
|---|---|---|---|
| `main` | Production-ready code | ‚Äî | ‚Äî |
| `develop` | Integration branch | `main` | `main` (via `release-*`) |
| `feature/*` | New work | `develop` | `develop` |
| `release/*` | Release stabilization | `develop` | `main` |
| `fix/*` | Bug fixes | `develop` | `develop` |

`main` and `develop` are protected. All other branches are ephemeral.

---

## Merge Rules

| Target | Allowed source | Merge type |
|---|---|---|
| `develop` | `feature/*`, `fix/*` | Squash merge |
| `main` | `release-*` only | Merge commit |

No direct commits to `main` or `develop`. Everything goes through a PR.

---

## Linear Git History

Enforced on both `main` and `develop` via GitHub Rulesets (`required_linear_history`).

Feature branches use squash merge into `develop`, which keeps the integration branch linear. Release branches use merge commits into `main` to preserve the release boundary.

---

## Branch Protection

Enforced via GitHub Rulesets on `main` and `develop`:

- **No direct push** ‚Äî all changes require a PR
- **No force push** (`non_fast_forward` rule) ‚Äî history is immutable on protected branches
- **No deletion** (`deletion` rule) ‚Äî `main` and `develop` cannot be deleted

Feature, release, and fix branches are not protected. They can be created, force-pushed, and deleted freely.

---

## PR Requirements

- PRs are required before merging into any protected branch
- **Required approving reviews**: set by `required_reviews` in `sentinel.yml` ‚Äî git-sentinel applies whatever the user configures (default: `0`)
- Stale reviews are dismissed on push (enabled on `main` ruleset)
- The tool does not hardcode a review count. Repository owners decide.

---

## Delete Branch on Merge

Controlled by `delete_branch_on_merge` in `sentinel.yml` (default: `true`).

When enabled, GitHub auto-deletes the source branch after a PR is merged. Applies to `feature/*`, `fix/*`, and `release/*` branches. Protected branches (`main`, `develop`) are never deleted regardless of this setting.

---

## Signed Commits

Encouraged for public repos. Not enforced by default.

Public repos benefit from verified commit identity, but enforcing signatures creates friction for new contributors. Repository owners can add signature requirements manually or via org-level rulesets if needed.

Out of scope for git-sentinel v1.

---

## CONTRIBUTING.md

Recommended for public repos. Not generated by git-sentinel.

Public repos should include a `CONTRIBUTING.md` that covers:
- How to report issues
- How to submit PRs
- Branch naming conventions (already documented in `GIT_REFERENCE.md`)
- Code style and testing expectations

Inject one via the `templates:` field in `sentinel.yml` if you have a standard contributing guide.

---

## No Secrets in Git History

git-sentinel does not scan for secrets, but public repos must treat this seriously:

- Use `.gitignore` to exclude sensitive files (`.env`, credentials, keys)
- Use environment variables or a secrets vault for runtime secrets
- Never commit API keys, tokens, passwords, or private keys
- If a secret is committed, rotate it immediately ‚Äî removing from history is not enough

## PR Template

git-sentinel generates `.github/PULL_REQUEST_TEMPLATE.md` with this structure:

```
## What
<!-- What does this PR do? -->

## Why
<!-- Why is this change needed? -->

## How
<!-- How was it implemented? Key decisions? -->

## Testing
<!-- How was it tested? -->

## Checklist
- [ ] Self-reviewed
- [ ] No debug/test code left in
- [ ] Relevant docs updated
```

This is tool-owned. Overwritten on `enforce`.

---

## GIT_REFERENCE.md

Auto-generated into every repo by git-sentinel. Contains:

- Branch strategy and merge rules
- Enforced rulesets and rationale
- Common git commands for daily workflow
- Recovery guide (restore deleted branch, undo commits, resolve conflicts)

This is tool-owned. Overwritten on `enforce`. Do not edit manually.
